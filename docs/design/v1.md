# Mullion v1 — Design

Status: living document. Source of truth for v1 scope and architecture.

## Goal

A macOS window-management app **designed for ultrawide and superwide monitors** (21:9, 32:9). Existing tools (Rectangle, Magnet, Loop, Moom) treat an ultrawide as one screen with halves and thirds. Mullion treats it as the primary surface and ships layouts, zones, and per-app rules that actually use the space.

macOS exposes an ultrawide as a single `NSScreen`. There is no supported way to make it behave as multiple displays (PbP needs two physical inputs; virtual-display tools are unreliable). Mullion provides its own zone system that *feels* like a multi-monitor setup — sticky per-zone focus, per-app default zones, hotkey snapping, optional gaps, asymmetric layouts.

The user-visible north star: a docked-undocked-docked cycle should never require touching a config file, and every primitive Mullion ships should have a UI surface — not just `layouts.json`.

## v1 scope

### Engine
- Zone engine — user-defined rectangular zones per display, saved as named layouts
- Layout-level gaps / outer margins (per-layout `outerMargin` + `innerGap`, applied by `FrameResolver`)
- Per-app default zones (explicit rules + learned history)
- Sticky per-zone focus (hotkey-to-focus + cycle ordering)
- Cycling on the same hotkey (single hotkey → ordered list of zones)
- Auto-restore window placements on launch (toggleable)
- Per-app `compatibilityProfile` — escape hatch for AX-resistant apps (Electron/Office)
- `SystemWindowManager` fallback path, gated behind `compatibilityProfile == .systemWindowManager` (half/third only)

### Hotkeys
- Global hotkeys to snap the focused window to a named zone
- Mouse-drag-to-snap preview overlay (release inside a zone → snap)
- Hold-modifier-to-show-grid overlay (configurable modifier mask)

### Multi-display
- Multi-display aware — laptop + ultrawide must not break
- Arrangement-as-unit configs — recognize a recurring display arrangement and bind a default layout to it

### Workspaces
- Workspaces — named tuple of `(layout, [(bundleID, zoneID)])`, optionally bound to an arrangement for auto-restore

### UI
- Menu-bar app with a layout picker
- Ultrawide-first preset pack — asymmetric zones, 6-pane, 1/4-1/2-1/4, center-stage + side rails
- SwiftUI layout editor with live aspect-correct preview (zones list + per-zone inspector with axis presets and fill-gap actions)
- App Rules editor UI (extends the layout editor window)
- Hotkey Bindings editor UI (extends the layout editor window; KeyboardShortcuts `Recorder` view)
- FSEvents-driven auto-reload of every JSON config file (manual "Reload" stays as a fallback)

### Permissions
- Clean Accessibility permission flow on first launch

### Out of scope for v1

- Tiling / automatic placement (yabai-style)
- Stage Manager integration (no public API)
- Cloud sync / multi-machine settings
- Touchbar / hardware-key surfaces
- Localization (English-only)
- Per-Space layouts (Mullion stays Space-agnostic)
- Plugin-style non-rectangular zone shapes
- Public AppleScript / URL-scheme surface

## Platform targets

- macOS 15.0 (Sequoia) minimum. No back-compat shims.
- Swift 5.10, Xcode 15+.
- AppKit (not SwiftUI for the menu-bar surface; SwiftUI acceptable for settings/onboarding/editor windows).
- Not sandboxed (AX API requires it).
- Distribution: direct download via GitHub Releases, Sparkle 2 for updates.

We stay AX-first as the placement engine even though Sequoia ships a native tiling API (`SystemWindowManager`). Loop's experience shows the native API doesn't express asymmetric / 1/4-1/2-1/4 / center-stage layouts well — exactly the layouts Mullion exists to support. `SystemWindowManager` is used only as the explicit per-app fallback for AX-resistant apps (half/third only).

Layouts, arrangements, workspaces, and compatibility profiles are all data. **Layouts are data, not code.**

### Key types

```swift
struct Zone: Codable, Identifiable {
    let id: UUID
    var name: String                  // "Center stage", "Left rail"
    var frame: CGRect                 // normalized 0…1 within display
    var anchor: Anchor                // for absolute-size zones
    var sizeOverride: CGSize?         // pixel-pinned width/height
}

struct Layout: Codable, Identifiable {
    let id: UUID
    var name: String                  // "Ultrawide 6-pane"
    var zones: [Zone]
    var displayPredicate: DisplayPredicate
    // .anyDisplay | .aspectRatio(min: Double) | .specificDisplay(uuid: String)
    var outerMargin: EdgeInsets       // 0 default; in display points
    var innerGap: Double              // 0 default; symmetric gap between adjacent zones
}

struct HotkeyBinding: Codable, Identifiable {
    let id: UUID
    var shortcutName: String          // KeyboardShortcuts.Name raw
    var targets: [UUID]               // Zone.IDs; len 1 = single, len >1 = cycle
    var role: Role                    // .snap | .focus
}

struct AppRule: Codable, Identifiable {
    let id: UUID
    var bundleID: String              // "com.tinyspeck.slackmacgap"
    var displayPredicate: DisplayPredicate
    var preferredZoneID: UUID
    var compatibilityProfile: CompatProfile  // .standard default
}

enum CompatProfile: String, Codable, Hashable {
    case standard               // current WindowMutator behavior
    case aggressive             // extra settle delay + post-write verify
    case systemWindowManager    // route through SystemWindowManager
}

struct LearnedPlacement: Codable {
    let bundleID: String
    let displayUUID: String
    var zoneID: UUID
    var placedAt: Date
}

struct Arrangement: Codable, Identifiable, Hashable {
    let id: UUID
    var name: String                  // "Home desk", "Laptop only"
    var signature: [DisplaySig]       // sorted; equality = arrangement identity
    var defaultLayoutID: UUID?        // applied on arrangement match
}

struct DisplaySig: Codable, Hashable {
    let displayUUID: String
    let widthPoints: Int              // rounded to 10pt to absorb scale noise
    let heightPoints: Int
    let originX: Int
    let originY: Int
}

struct Workspace: Codable, Identifiable {
    let id: UUID
    var name: String
    var layoutID: UUID
    var placements: [WorkspacePlacement]
    var arrangementID: UUID?          // optional: auto-apply on arrangement match
}

struct WorkspacePlacement: Codable, Hashable {
    let bundleID: String
    let zoneID: UUID
    let windowTitlePattern: String?   // optional regex to disambiguate multi-window apps
}

struct AppSettings: Codable {
    var autoRestoreEnabled: Bool = true
}
```

### Placement resolution

When deciding where a window should go (auto-restore on launch, or "place this app" command):

1. Match `AppRule` for `(bundleID, displayPredicate)` → use it
2. Else `LearnedPlacement` for `(bundleID, displayUUID)` → use it
3. Else no-op (user moves manually, which becomes the next `LearnedPlacement`)

Explicit rules always beat learned history.

### Cycling

A `HotkeyBinding` carries `targets: [Zone.ID]`. Pressing the hotkey:

- `targets.count == 1` → snap to that zone
- `targets.count > 1` → snap to next zone in the cycle, tracked per (window, binding) so each window cycles independently

### Focus index (`.focus` role)

`FocusIndex` is a `[ZoneID: [AXWindowSignature]]` MRU list. Updates:

- Every successful `.snap` appends/promotes the window in the target zone.
- `kAXFocusedWindowChangedNotification` on each running app promotes whichever zone currently contains the window (computed by overlap).
- On window-closed AX notification, evict.

`.focus` dispatch: look up the zone's MRU list, raise the most-recent still-alive window. If empty, fall back to focusing the app whose `AppRule` points at that zone.

### Arrangement resolution

On `DisplayRegistry.onChange`:

1. Build current `signature: [DisplaySig]` from `NSScreen.screens`.
2. Look up `Arrangement` by exact signature match. If found and it has a `defaultLayoutID`, apply that layout.
3. Else publish "unknown arrangement" — UI offers "Save current as arrangement…" affordance in the menu-bar dropdown.

Signature matching is exact on rounded points; we deliberately do not fuzzy-match because two distinct docking arrangements can have very similar geometry.

### Drag/grid overlay tap

One `CGEventTap(.listenOnly)` mounted at the session level. Multiplexed:

- **Drag**: enabled while `leftMouseDown` is held; renders the `DragOverlayController` window with the zone under the cursor highlighted; `leftMouseUp` inside a zone triggers a snap.
- **Grid**: enabled while the configured modifier mask is held without any mouse button; renders the `GridOverlayController` with every zone outlined.

Both overlays are borderless `NSWindow`s with `level = .floating`, `ignoresMouseEvents = true`, `collectionBehavior = [.canJoinAllSpaces, .stationary, .fullScreenAuxiliary]`. One window per display. Mouse events go through the AX permission Mullion already holds; Input Monitoring is not required.

## Persistence

All in `~/Library/Application Support/Mullion/`:

- `layouts.json` — `[Layout]` (includes `outerMargin` + `innerGap`)
- `bindings.json` — `[HotkeyBinding]`
- `app-rules.json` — `[AppRule]` (includes `compatibilityProfile`)
- `window-history.json` — `[LearnedPlacement]`, authoritative for auto-restore
- `arrangements.json` — `[Arrangement]`
- `workspaces.json` — `[Workspace]`
- `settings.json` — `AppSettings`

Schema-versioned (`{ "version": 1, ... }`). Atomic writes, 500ms debounce. Plain `JSONEncoder`/`Decoder`. One file per concern, `cat`-inspectable.

FSEvents watches all of the above; debounce 250ms; re-publish through the relevant store. The "Reload Layouts" menu item remains as a manual fallback.

Default ultrawide preset pack ships bundled in `Resources/DefaultLayouts.json` and seeds `layouts.json` on first run.

## Accessibility permission gate

Loop's event-driven approach plus a yabai-style live probe (because `AXIsProcessTrusted()` lies after some macOS updates).

**First launch:**

1. Call `AXIsProcessTrusted()` without prompt.
2. If false: show custom `OnboardingWindow` (explanation + screenshot + "Open System Settings" deep link `x-apple.systempreferences:com.apple.preference.security?Privacy_Accessibility` + "Quit & Reopen" button).
3. Subscribe to `DistributedNotificationCenter` on `com.apple.accessibility.api`. On notification, re-check; dismiss window on grant.

**Runtime revocation detection:**

- Trust `AXIsProcessTrusted` for the launch gate.
- On any `kAXErrorCannotComplete` from a mutation: kick a `CGEventTapCreate(.listenOnly)` probe. If the probe fails, surface a re-auth banner in the menu-bar dropdown. Auto-offer "Quit & Reopen" after the second failure within a session.
- Never modal-block the user during use.

## Hotkeys

[KeyboardShortcuts by Sindre Sorhus](https://github.com/sindresorhus/KeyboardShortcuts) (MASShortcut successor). Under the hood uses Carbon `RegisterEventHotKey`, which is the only path that does **not** trigger the Input Monitoring permission.

Tradeoffs accepted: no modifier-only chords, no key-up events. The hold-modifier-to-show-grid feature does its modifier-tracking via the shared `CGEventTap` instead — the AX permission Mullion already holds covers it.

## Signing & notarization

- Apple Developer Program ($99/yr), Developer ID Application cert
- Hardened runtime enabled. **No special entitlements** — AX is TCC-gated, not entitlement-gated
- `LSUIElement = YES`, activation policy `.accessory`
- Build pipeline: `archive` → `xcrun notarytool submit --wait` (credentials stored via `notarytool store-credentials` from an App Store Connect API key) → `xcrun stapler staple`
- Updates: Sparkle 2. Sign Sparkle framework individually before the app bundle. Never `codesign --deep`. For non-sandboxed Mullion, drop Sparkle's bundled XPC services entirely

## Top risks and mitigations

1. **`AXEnhancedUserInterface` jitter on Electron + Office.** Some apps (MS Office, older Electron) enable `AXEnhancedUserInterface` when an automation client attaches; with it set, every position/size change is animated and frames don't land cleanly. **Mitigation:** every position/size mutation goes through `WindowMutator`, which reads → clears → writes → restores `AXEnhancedUserInterface` (yabai's `AX_ENHANCED_UI_WORKAROUND` macro pattern). Per-app `compatibilityProfile` field (`.standard | .aggressive | .systemWindowManager`) gives an explicit escape hatch when the global mitigation isn't enough.

2. **AX revocation lying via stale cache.** `AXIsProcessTrusted()` can return `true` while every mutation returns `kAXErrorCannotComplete`. **Mitigation:** dual signal — boolean for the gate, `kAXErrorCannotComplete` triggers a `CGEventTap(.listenOnly)` probe and surfaces an in-app banner. Auto-offer "Quit & Reopen" after the second failure.

3. **Multi-display coordinate math.** AppKit uses bottom-left origin; AX uses top-left global coordinates. Negative offsets for monitors above/left of primary. `NSScreen.main` is the key window's screen, not the origin anchor — using it is a classic off-by-display bug that *only* manifests for users with non-trivial arrangements (i.e., the target market). **Mitigation:** one `Geometry` module is the sole conversion site, anchored on `NSScreen.screens.first { $0.frame.origin == .zero }`. Fixture-based unit tests for primary-only, left-secondary (negative-x), top-secondary (negative-y), four-corner.

4. **Display identity across reconnect.** `CGDirectDisplayID` is not stable across reconnect; layouts keyed on it scramble on lid open. **Mitigation:** persist `CGDisplayCreateUUIDFromDisplayID` string. Fall back to `(localizedName, frame.size)` for identical-twin panels. Debounce `didChangeScreenParametersNotification` 250ms.

5. **Full-screen windows ignore AX.** Native-fullscreen windows (`kAXFullscreenAttribute = true`) silently no-op size/position writes. **Mitigation:** read the attribute and clear it before the mutation. Stage Manager has no public API — accept as out-of-scope; document.

6. **`CGEventTap` revocation.** Same TCC family as AX, but a separate approval surface in some macOS revisions. **Mitigation:** lazy-create the tap; on creation failure, log + disable drag/grid features gracefully and surface a one-line banner. Never crash, never block the rest of the app — engine features must keep working even if the tap refuses to mount.

7. **Arrangement signature collisions.** Two physically different arrangements can produce identical signatures if displays report the same point dimensions and origins (common with identical-twin panels reconnecting in a different physical position). **Mitigation:** include `displayUUID` in the signature so identical panels still distinguish. For displays without UUIDs, fall back to `(localizedName, size)` as the rest of the app already does.

8. **Workspace restore races a not-yet-launched app.** Capturing a workspace that included an app that's now quit means there's no window to place. **Mitigation:** workspace restore is best-effort: missing apps are reported via a transient menu-bar banner with "Launch missing apps" action; no error dialogs. We do not auto-launch — the user opted out by quitting.

9. **Overlay flicker during fast drags.** Repositioning a borderless `NSWindow` every mouse-move event is expensive. **Mitigation:** keep one overlay window per display, persistent across the drag; mutate its highlight rectangle only when the cursor crosses a zone boundary, not on every event. Coalesce events through a throttled main-queue dispatch.

10. **`.focus` MRU drift.** AX focus notifications are not 100% reliable — some apps fire on window activation, some don't. **Mitigation:** the MRU list is advisory, not authoritative. On `.focus` dispatch, validate the candidate window still exists and is in the expected zone (overlap ≥ 50%); skip and try the next entry if not.

## Build order

Each step delivers something observable. Phase 0 is shipped; the rest is the remaining v1 work in execution order.

### Phase 0 — shipped

1. **AX check.** `AccessibilityGate.isTrusted` exists. App boots, logs `AXIsProcessTrusted()` to console. Manual grant via System Settings.
2. **Focused window resolution.** `FocusedWindow.current()` returns the frontmost window's `AXUIElement`. Smoke test by logging its title.
3. **Window mutation.** `WindowMutator.set(window, frame:)` implements size→position→size + `AXEnhancedUserInterface` toggle. Hardcode `CGRect(100, 100, 800, 600)`. Confirm a real window moves.
4. **Geometry + multi-display tests.** `Geometry.appKitToAX(_:)` anchored on origin-zero screen. Fixture tests for all four arrangement classes. `CGRect(0.25, 0, 0.5, 1)` of a secondary `NSScreen` lands correctly.
5. **Hotkey binding.** KeyboardShortcuts wired. ⌃⌥⌘← snaps focused window to left half of its display.
6. **Layout/Zone/LayoutStore.** Codable JSON catalogs with three zones, one hotkey per zone.
7. **Status item.** Menu-bar item with layout list.
8. **Onboarding window + revoke listener.** `DistributedNotificationCenter` subscription. Onboarding shows on missing permission, dismisses on grant. Re-onboards when a hotkey fires without AX trust.
9. **Cycle behavior.** `HotkeyBinding.targets` cycles through multiple zones per binding.
10. **App rules + Reload Layouts menu.** Explicit `AppRule` lookup before dispatch. Reload picks up edits to the JSON files.
11. **Learned placements.** Writeback on every snap. Resolution path 1→2→3.
12. **Auto-restore on launch.** Walk current windows, apply `LearnedPlacement` where available. Toggle via menu (`AppSettings.autoRestoreEnabled`).
13. **Ultrawide preset pack.** Bundled `DefaultLayouts.json` with asymmetric / 6-pane / 1/4-1/2-1/4 / center-stage layouts. Seeded on first run.
14. **SwiftUI layout editor.** Live aspect-correct preview, axis presets, fill-gap actions in the zone inspector.

### Phase A — finish v1 stubs

15. **`.focus` role.** Implement `FocusIndex`. `.focus` hotkeys raise the most-recent window in the target zone. Tests: round-trip `snap` → `focus` on fixture windows. Closes a v1-scope item that shipped as a stub.
16. **Layout `outerMargin` + `innerGap`.** Additive fields, `FrameResolver` honors them, layout editor exposes two number inputs in the inspector.

### Phase B — release readiness

17. **Sparkle + notarization.** CI pipeline for signed, notarized, stapled DMG with Sparkle feed. Pulled forward from the original "last step" position: shipping at the smallest possible scope unlocks the real-user feedback that should shape Phases D–F. Mechanical work, no design risk. **First public-beta milestone.**

### Phase C — kill JSON-editing friction

18. **App Rules editor UI.** Extend `LayoutEditorWindow` with a second sidebar section; same JSON-source-of-truth model the layouts editor uses today.
19. **Bindings editor UI.** Third sidebar section. KeyboardShortcuts' `Recorder` view drops directly in.
20. **FSEvents auto-reload.** Watch the config files; debounce 250ms; re-emit through the relevant stores. Keep manual "Reload" as fallback.
21. **`compatibilityProfile` field on `AppRule`.** `WindowMutator` consults it for the aggressive-settle path. UI: a popup in the App Rules editor. Pairs with #18 — ship together.

### Phase D — arrangement story

22. **Arrangement detection.** `ArrangementRegistry` derives signature, matches against `arrangements.json`. Menu-bar banner: "Unknown arrangement — Save as…".
23. **Arrangement → default layout.** When matched, apply `defaultLayoutID`. **First end-to-end arrangement-as-unit milestone.**

### Phase E — mouse-driven UX

24. **Mouse event tap.** Mount the shared `CGEventTap`; verify left-mouse events arrive without breaking input. No UI yet. Foundation for #25 and #26.
25. **Drag-to-snap overlay.** Render highlight on hover, snap on release.
26. **Hold-modifier grid overlay.** Reuse overlay machinery. Configurable modifier mask in settings.

### Phase F — workspaces

27. **Workspaces — capture + restore.** UI in the editor window. No arrangement binding yet.
28. **Workspaces — arrangement binding.** Auto-restore a workspace on arrangement match.

### Phase G — exploratory, conditional

29. **`SystemWindowManager` fallback.** Gated entirely behind `compatibilityProfile == .systemWindowManager`. Half/third support only. Build only if a real user reports an app where the AX path can't be made to work.

Steps 1–14 are shipped. Phase A finishes the two outstanding v1-scope items. Phase B is the leverage move — every feature past this point ships to actual users instead of into a vacuum. Phases C–F build out the editor / arrangement / mouse / workspace surfaces in roughly that order. Phase G is conditional on real-world demand.

## Resolved open questions

| Question | Decision |
|---|---|
| Zone editing UX | SwiftUI editor with live aspect-correct preview + raw-JSON fallback via "Reload" menu. |
| Per-app default zone | Both: explicit `AppRule` wins, `LearnedPlacement` is the fallback. |
| Sticky per-zone focus | Both: hotkey-to-focus (via `FocusIndex` MRU) AND cycle ordering. |
| Cycle on same hotkey | Yes. `HotkeyBinding.targets` is an ordered list. |
| macOS deployment target | 15.0 minimum, no back-compat. |
| AX-only vs SystemWindowManager | AX-first; `SystemWindowManager` as the explicit per-app fallback gated by `compatibilityProfile`. |
| Per-display vs per-arrangement layouts | Both: per-display zones, plus arrangement-as-unit defaults via `Arrangement.defaultLayoutID`. |
| Auto-restore on launch | Default on, toggleable via `AppSettings.autoRestoreEnabled`. |
| Config editing | All five JSON files have UI editors; FSEvents auto-reload picks up direct edits. |
| Workspaces | In scope. Distinct from `LearnedPlacement` (history) and `AppRule` (per-app default) — a workspace captures a specific *moment*. |
| Distribution | Direct download + Sparkle, GitHub Releases as the feed. |

## Future work (post-v1)

- Tiling-mode / yabai-style automatic placement, gated behind an opt-in flag — would require rethinking `LearnedPlacement` semantics
- Multi-display arrangement *templates* (auto-generate a layout per arrangement based on aspect ratio classes)
- Per-Space layouts (currently Mullion is Space-agnostic)
- Plugin-style custom zone shapes (non-rectangular) — almost certainly a bad idea but worth naming so it can be explicitly declined
- Public AppleScript / URL-scheme surface for scripting layouts
- Companion CLI (`mullion snap left-rail`) for users who want it
- Settings sync via iCloud Drive (single-machine continues to be the supported story; sync is opt-in only if it ever ships)
