# Mullion v1 — Design

Status: locked. This document is the source of truth for v1 scope and architecture.

## Goal

A macOS window-management app **designed for ultrawide and superwide monitors** (21:9, 32:9). Existing tools (Rectangle, Magnet, Loop, Moom) treat an ultrawide as one screen with halves and thirds. Mullion treats it as the primary surface and ships layouts, zones, and per-app rules that actually use the space.

macOS exposes an ultrawide as a single `NSScreen`. There is no supported way to make it behave as multiple displays (PbP needs two physical inputs; virtual-display tools are unreliable). Mullion provides its own zone system that *feels* like a multi-monitor setup — sticky per-zone focus, per-app default zones, hotkey snapping, optional gaps, asymmetric layouts.

## v1 scope

- Zone engine — user-defined rectangular zones per display, saved as named layouts
- Global hotkeys to snap the focused window to a named zone
- Menu-bar app with a layout picker
- Ultrawide-first preset pack — asymmetric zones, 6-pane, 1/4-1/2-1/4, center-stage + side rails
- Multi-display aware — laptop + ultrawide must not break
- Clean Accessibility permission flow on first launch
- Per-app default zones (explicit rules + learned history)
- Sticky per-zone focus (hotkey-to-focus + cycle ordering)
- Cycling on the same hotkey (single hotkey → ordered list of zones)
- Auto-restore window placements on launch (toggleable)

### Out of scope for v1

- Tiling / automatic placement (yabai-style)
- Workspaces
- Stage Manager integration
- Mouse-drag zone preview UI
- Layout editor UI (raw JSON + Reload Layouts menu instead)
- Arrangement-as-unit configs (per-display only in v1; arrangement-aware as v2)
- FSEvents-driven layout auto-reload (menu-driven only in v1)

## Platform targets

- macOS 15.0 (Sequoia) minimum. No back-compat shims.
- Swift 5.10, Xcode 15+.
- AppKit (not SwiftUI for the menu-bar surface; SwiftUI acceptable for settings/onboarding windows).
- Not sandboxed (AX API requires it).
- Distribution: direct download via GitHub Releases, Sparkle 2 for updates.

We stay AX-first as the placement engine even though Sequoia ships a native tiling API (`SystemWindowManager`). Loop's experience shows the native API doesn't express asymmetric / 1/4-1/2-1/4 / center-stage layouts well — exactly the layouts Mullion exists to support. SystemWindowManager remains available as a fallback for apps that resist AX; reevaluate after v1.

## Architecture

### Module breakdown

```
Mullion/
  Core/
    main.swift                   — entry point
    AppDelegate.swift            — lifecycle
    StatusItemController.swift   — NSStatusItem + dropdown menu
  Window/
    AXWindow.swift               — wrapper around AXUIElement
    FocusedWindow.swift          — frontmost PID → AX app → focused window
    WindowMutator.swift          — size→position→size + AXEnhancedUserInterface toggle
    WindowMover.swift            — mover protocol; Standard / BestEffort / CenteringFixedSize
    WindowHistory.swift          — authoritative placement record per (displayUUID, bundleID)
  Layout/
    Zone.swift                   — normalized CGRect + anchor + optional pixel override
    Layout.swift                 — name + [Zone] + display predicate
    LayoutStore.swift            — Codable JSON persistence
    LayoutResolver.swift         — selects active layout per display
    FrameResolver.swift          — Zone × NSScreen → CGRect
  Hotkeys/
    HotkeyManager.swift          — wraps KeyboardShortcuts (Sindre Sorhus)
    ActionDispatcher.swift       — hotkey → focused window → zone(s) → mover
  Display/
    Geometry.swift               — single AppKit↔AX conversion site
    DisplayRegistry.swift        — CGDisplayUUID lookup, hot-swap debounce
  Permissions/
    AccessibilityGate.swift      — check, prompt, live probe
  Rules/
    AppRule.swift                — explicit per-app zone preference
    AppRuleStore.swift           — Codable JSON persistence
    LearnedPlacement.swift       — auto-memory of last placement per app+display
  UI/
    LayoutPickerMenu.swift       — dropdown contents
    OnboardingWindow.swift       — first-run AX flow
  Permissions/
    AccessibilityGate.swift
  Resources/
    Info.plist
    Mullion.entitlements
    DefaultLayouts.json          — bundled ultrawide preset pack
```

Roughly 20 source files at v1 completion. Same scale as Rectangle's core minus its 90-file `WindowCalculation/` directory — which Mullion deliberately does not replicate. **Layouts are data, not code.**

### Key types

```swift
struct Zone: Codable, Identifiable {
    let id: UUID
    var name: String                  // "Center stage", "Left rail"
    var frame: CGRect                 // normalized 0…1 within display
    var anchor: Anchor                // for absolute-size zones
    var sizeOverride: CGSize?         // pixel-pinned width/height
}

struct Layout: Codable, Identifiable {
    let id: UUID
    var name: String                  // "Ultrawide 6-pane"
    var zones: [Zone]
    var displayPredicate: DisplayPredicate
    // .anyDisplay | .aspectRatio(min: Double) | .specificDisplay(uuid: String)
}

struct HotkeyBinding: Codable, Identifiable {
    let id: UUID
    var shortcutName: String          // KeyboardShortcuts.Name raw
    var targets: [UUID]               // Zone.IDs; len 1 = single, len >1 = cycle
    var role: Role                    // .snap | .focus
}

struct AppRule: Codable, Identifiable {
    let id: UUID
    var bundleID: String              // "com.tinyspeck.slackmacgap"
    var displayPredicate: DisplayPredicate
    var preferredZoneID: UUID
}

struct LearnedPlacement: Codable {
    let bundleID: String
    let displayUUID: String
    var zoneID: UUID
    var placedAt: Date
}

struct AppSettings: Codable {
    var autoRestoreEnabled: Bool = true
}
```

### Placement resolution

When deciding where a window should go (auto-restore on launch, or "place this app" command):

1. Match `AppRule` for `(bundleID, displayPredicate)` → use it
2. Else `LearnedPlacement` for `(bundleID, displayUUID)` → use it
3. Else no-op (user moves manually, which becomes the next `LearnedPlacement`)

Explicit rules always beat learned history.

### Cycling

A `HotkeyBinding` carries `targets: [Zone.ID]`. Pressing the hotkey:

- `targets.count == 1` → snap to that zone
- `targets.count > 1` → snap to next zone in the cycle, tracked per (window, binding) so each window cycles independently

## Persistence

All in `~/Library/Application Support/Mullion/`:

- `layouts.json` — `[Layout]`
- `bindings.json` — `[HotkeyBinding]`
- `app-rules.json` — `[AppRule]`
- `window-history.json` — `[LearnedPlacement]`, authoritative for auto-restore
- `settings.json` — `AppSettings`

Schema-versioned (`{ "version": 1, ... }`). Atomic writes, 500ms debounce. Plain `JSONEncoder`/`Decoder`. One file per concern, `cat`-inspectable.

Default ultrawide preset pack ships bundled in `Resources/DefaultLayouts.json` and seeds `layouts.json` on first run.

## Accessibility permission gate

Loop's event-driven approach plus a yabai-style live probe (because `AXIsProcessTrusted()` lies after some macOS updates).

**First launch:**

1. Call `AXIsProcessTrusted()` without prompt.
2. If false: show custom `OnboardingWindow` (explanation + screenshot + "Open System Settings" deep link `x-apple.systempreferences:com.apple.preference.security?Privacy_Accessibility` + "Quit & Reopen" button).
3. Subscribe to `DistributedNotificationCenter` on `com.apple.accessibility.api`. On notification, re-check; dismiss window on grant.

**Runtime revocation detection:**

- Trust `AXIsProcessTrusted` for the launch gate.
- On any `kAXErrorCannotComplete` from a mutation: kick a `CGEventTapCreate(.listenOnly)` probe. If the probe fails, surface a re-auth banner in the menu-bar dropdown. Auto-offer "Quit & Reopen" after the second failure within a session.
- Never modal-block the user during use.

## Hotkeys

[KeyboardShortcuts by Sindre Sorhus](https://github.com/sindresorhus/KeyboardShortcuts) (MASShortcut successor). Under the hood uses Carbon `RegisterEventHotKey`, which is the only path that does **not** trigger the Input Monitoring permission.

Tradeoffs accepted: no modifier-only chords, no key-up events. If Mullion ever needs hold-modifier-to-show-grid UX, add a `CGEventTap` alongside the Carbon registration; the AX permission Mullion already holds covers it.

## Signing & notarization

- Apple Developer Program ($99/yr), Developer ID Application cert
- Hardened runtime enabled. **No special entitlements** — AX is TCC-gated, not entitlement-gated
- `LSUIElement = YES`, activation policy `.accessory`
- Build pipeline: `archive` → `xcrun notarytool submit --wait` (credentials stored via `notarytool store-credentials` from an App Store Connect API key) → `xcrun stapler staple`
- Updates: Sparkle 2. Sign Sparkle framework individually before the app bundle. Never `codesign --deep`. For non-sandboxed Mullion, drop Sparkle's bundled XPC services entirely

## Top risks and mitigations

1. **`AXEnhancedUserInterface` jitter on Electron + Office.** Some apps (MS Office, older Electron) enable `AXEnhancedUserInterface` when an automation client attaches; with it set, every position/size change is animated and frames don't land cleanly. **Mitigation:** every position/size mutation goes through `WindowMutator`, which reads → clears → writes → restores `AXEnhancedUserInterface` (yabai's `AX_ENHANCED_UI_WORKAROUND` macro pattern). Per-app `compatibilityProfile` field for apps that still misbehave.

2. **AX revocation lying via stale cache.** `AXIsProcessTrusted()` can return `true` while every mutation returns `kAXErrorCannotComplete`. **Mitigation:** dual signal — boolean for the gate, `kAXErrorCannotComplete` triggers a `CGEventTap(.listenOnly)` probe and surfaces an in-app banner. Auto-offer "Quit & Reopen" after the second failure.

3. **Multi-display coordinate math.** AppKit uses bottom-left origin; AX uses top-left global coordinates. Negative offsets for monitors above/left of primary. `NSScreen.main` is the key window's screen, not the origin anchor — using it is a classic off-by-display bug that *only* manifests for users with non-trivial arrangements (i.e., the target market). **Mitigation:** one `Geometry` module is the sole conversion site, anchored on `NSScreen.screens.first { $0.frame.origin == .zero }`. Fixture-based unit tests for primary-only, left-secondary (negative-x), top-secondary (negative-y), four-corner.

4. **Display identity across reconnect.** `CGDirectDisplayID` is not stable across reconnect; layouts keyed on it scramble on lid open. **Mitigation:** persist `CGDisplayCreateUUIDFromDisplayID` string. Fall back to `(localizedName, frame.size)` for identical-twin panels. Debounce `didChangeScreenParametersNotification` 250ms.

5. **Full-screen windows ignore AX.** Native-fullscreen windows (`kAXFullscreenAttribute = true`) silently no-op size/position writes. **Mitigation:** read the attribute and clear it before the mutation. Stage Manager has no public API — accept as out-of-scope; document.

## Build order

Each step delivers something observable.

1. **AX check.** `AccessibilityGate.isTrusted` exists. App boots, logs `AXIsProcessTrusted()` to console. Manual grant via System Settings.
2. **Focused window resolution.** `FocusedWindow.current()` returns the frontmost window's `AXUIElement`. Smoke test by logging its title.
3. **Window mutation.** `WindowMutator.set(window, frame:)` implements size→position→size + `AXEnhancedUserInterface` toggle. Hardcode `CGRect(100, 100, 800, 600)`. Confirm a real window moves.
4. **Geometry + multi-display tests.** `Geometry.appKitToAX(_:)` anchored on origin-zero screen. Fixture tests for all four arrangement classes. Now `CGRect(0.25, 0, 0.5, 1)` of a secondary `NSScreen` lands correctly. **First "snap one window" milestone.**
5. **Hotkey binding.** KeyboardShortcuts wired. ⌃⌥⌘← snaps focused window to left half of its display. Still no UI.
6. **Layout/Zone/LayoutStore.** Hardcoded `layouts.json` with three zones, one hotkey per zone.
7. **Status item.** Menu-bar item with static layout list. First real UI.
8. **Onboarding window + revoke listener.** `DistributedNotificationCenter` subscription. Onboarding shows on missing permission, dismisses on grant.
9. **Cycle behavior.** `HotkeyBinding.targets` cycles through multiple zones per binding.
10. **App rules + Reload Layouts menu.** Explicit `AppRule` lookup before dispatch. Reload picks up edits to `layouts.json` / `bindings.json` / `app-rules.json`.
11. **Learned placements.** Writeback on every snap. Resolution path 1→2→3.
12. **Auto-restore on launch.** Walk current windows, apply `LearnedPlacement` where available. Toggle via menu (`AppSettings.autoRestoreEnabled`).
13. **Ultrawide preset pack.** Bundled `DefaultLayouts.json` with asymmetric / 6-pane / 1/4-1/2-1/4 / center-stage layouts. Seeded on first run.
14. **Sparkle + notarization.** CI pipeline for signed, notarized, stapled DMG with Sparkle feed.

Steps 1–5 produce a working window-snapper in roughly a week. Everything after is product surface.

## Resolved open questions

| Question | Decision |
|---|---|
| Zone editing UX in v1 | Raw JSON + "Reload Layouts" menu item. No SwiftUI editor. |
| Per-app default zone | Both: explicit `AppRule` wins, `LearnedPlacement` is the fallback. |
| Sticky per-zone focus | Both: hotkey-to-focus AND cycle ordering. |
| Cycle on same hotkey | Yes. `HotkeyBinding.targets` is an ordered list. |
| macOS deployment target | 15.0 minimum, no back-compat. |
| AX-only vs SystemWindowManager | AX-first; SystemWindowManager reserved as a future fallback. |
| Per-display vs per-arrangement layouts | Per-display in v1, arrangement-as-unit in v2. Leave a hook on `DisplayRegistry`. |
| Auto-restore on launch | Default on, toggleable via `AppSettings.autoRestoreEnabled`. |
| Distribution | Direct download + Sparkle, GitHub Releases as the feed. |
